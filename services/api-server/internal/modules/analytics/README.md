# analytics

当前已落地：

- 审计日志列表接口
- 基础运营看板接口
- 基于统一 `audit` 存储的 `MongoDB` 主链路实现
- 已接入登录、教务绑定、课程表/考试/成绩/学期查询、门户公告发布、站内通知发布/已读、校园生活内容发布、审核更新等关键成功动作

约束：

- 当前分析能力只做后端事实汇总，不由前端自行拼接审计语义
- 管理读取权限以后端 `analytics:view` 为准
- 审计记录默认采用 best effort 写入，不阻塞主业务成功返回

当前边界：

- 当前审计日志已切到 `MongoDB`，但尚未补更细粒度埋点、指标与追踪
- 审计看板目前仍由读时聚合生成，尚未演进为预计算统计表

目标重构方向（设计阶段）：

- 审计日志、业务日志、状态迁移日志统一迁移到 `MongoDB`
- 审计记录按“事件事实 + 上下文快照”建模，不继续沿用简单关系型审计表思路
- 后续看板优先基于 `MongoDB` 读模型或离线聚合生成，但权限裁决仍在后端服务层完成
