# 0001 基础工程与架构落地计划

## 目标

把仓库从“空目录”推进到“可持续协作的工程底座”。

## 工作流

### 工作流 A：后端基础骨架

- 初始化 `services/api-server`
- 建立模块目录和分层约束
- 接入配置、日志、错误码、鉴权中间件

### 工作流 B：契约与代码生成

- 建立 `packages/contracts/openapi`
- 约定错误码、分页、鉴权、上传等通用 schema
- 设计 JS SDK 与 Dart SDK 生成流程

### 工作流 C：管理员后台外壳

- 初始化 `Vue 3 + Vite + TypeScript`
- 建立路由、布局、权限、SDK 调用骨架

### 工作流 D：微信小程序外壳

- 初始化原生小程序工程
- 建立登录、请求、分包、公共组件基础能力

### 工作流 E：Flutter App 外壳

- 初始化 `Flutter`
- 建立 `feature-first + MVVM` 目录
- 接入路由、状态管理、网络层和本地缓存抽象

### 工作流 F：质量与运维

- 建立 CI
- 接入 lint / test / format
- 预埋日志、指标、追踪、审计规范

## 里程碑

| 里程碑 | 说明 | 完成标志 |
| --- | --- | --- |
| `M0` | 文档与骨架完成 | 目录、规范、计划、架构文档齐备 |
| `M1` | 基础域可联调 | 认证、权限、门户、通知通路可用 |
| `M2` | 核心业务可试运行 | 课程、活动、工单具备可演示流程 |

## 风险

- 外部校园系统接口不稳定
- 多端并行开发导致接口漂移
- 权限模型在后期返工成本高
- 小程序和 Flutter 的体验诉求不同，容易错误共享 UI 思路

## 风险应对

- 所有外部系统先做 provider 封装与 mock
- 契约变更必须走版本和评审
- 权限模型先建资源动作码，再做界面
- 多端共享业务语义，不共享页面实现
