# 0001 基础工程与架构落地计划

## 目标

把仓库从“空目录”推进到“可持续协作的工程底座”。

## 工作流

### 工作流 A：后端基础骨架

- 初始化 `services/api-server`
- 建立模块目录和分层约束
- 接入配置、日志、错误码、鉴权中间件

### 工作流 B：契约与代码生成

- 建立 `packages/contracts/openapi`
- 约定错误码、分页、鉴权、上传等通用 schema
- 设计 JS SDK 与 Dart SDK 生成流程

### 工作流 C：管理员后台外壳

- 初始化 `Vue 3 + Vite + TypeScript`
- 建立路由、布局、权限、SDK 调用骨架

### 工作流 D：微信小程序外壳

- 初始化原生小程序工程
- 建立微信登录、请求、分包、公共组件基础能力
- 建立教务绑定状态感知与联系方式可见性控制的基础能力

当前进展（`2026-05-09`）：

- 已把核心目录职责收敛为 `api/` 传输层、`services/` 业务语义层、`stores/` 会话与配置层、`pages/` 展示层。
- 已重构微信登录、个人中心、教务绑定、首页动态、二手/跑腿/资料/失物招领列表页的数据流。
- 已补充小程序依赖接口文档，后续需继续与 `packages/contracts` 中的正式契约对齐。

当前进展（`2026-05-10`）：

- 已初始化 `services/api-server`，建立 `Go + Gin` 模块化单体基础骨架。
- 已落地配置、结构化日志、请求 ID、统一错误响应、鉴权上下文中间件。
- 已提供 `/healthz`、`/readyz`、`/api/v1/system/profile` 基础接口，并补齐最小 OpenAPI 契约基线。

### 工作流 E：Flutter App 外壳

- 初始化 `Flutter`
- 建立 `feature-first + MVVM` 目录
- 接入路由、状态管理、网络层和本地缓存抽象
- 对齐校园生活主线能力与教务绑定后的受限字段语义

### 工作流 F：质量与运维

- 建立 CI
- 接入 lint / test / format
- 预埋日志、指标、追踪、审计规范

## 风险

- 外部校园系统接口不稳定
- 多端并行开发导致接口漂移
- 权限模型在后期返工成本高
- 小程序和 Flutter 的体验诉求不同，容易错误共享 UI 思路
- 教务绑定状态与联系方式查看权限容易出现前后端语义不一致

## 风险应对

- 所有外部系统先做 provider 封装与 mock
- 契约变更必须走版本和评审
- 权限模型先建资源动作码，再做界面
- 多端共享业务语义，不共享页面实现
- 联系方式字段由后端统一裁剪，多端只消费可见性结果，不自行推断
