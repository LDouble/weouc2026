# 0004 存储分层与状态驱动重构计划

## 目标

把后端从当前的 `PostgreSQL + Redis` 主方案，演进为目标架构 `MySQL + MongoDB + Redis + BMFS`，并基于业务属性重建社区领域模型与统一状态驱动能力。

## 背景

- 当前 `iam` 已以 `PostgreSQL + Redis` 持久化
- 当前 `campus_life`、`portal`、`notification`、`analytics` 已围绕 `memory / postgres` 双后端实现
- 当前状态流转仍主要依赖各模块 `service` 中的显式条件判断
- 用户要求：
  - 社区相关内容、审计、日志、配置使用 `MongoDB`
  - 核心主数据使用 `MySQL`
  - `MySQL` 访问统一使用 `GORM`
  - 使用 `BMFS` 做状态驱动
  - 不完全复用当前数据结构
  - 审核态并入统一 `status`，不单独设计 `review_status`
  - 历史数据与旧链路可以不管
  - 历史代码逻辑可以删除，不保留

## 优先级

### P0：先把设计事实源立住

- 新增任务与 PRD
- 更新 `ARCHITECTURE.md`
- 更新 `docs/PLANS.md`
- 更新 `services/api-server` 与相关模块 README
- 明确 `BMFS` 的接入位与状态职责

### P1：建立迁移缝隙

- 在 `config/repo` 层增加 `mysql`、`mongodb` 后端抽象
- 梳理哪些读写要留在 `MySQL`
- 梳理哪些聚合和日志要迁到 `MongoDB`
- 为 `contracts` 设计新的单一 `status` 语义
- 建立查询白名单、输入校验、限流和安全错误响应基线
- 落地 `GORM` MySQL 客户端和仓储骨架

### P2：分阶段切流

- `iam` 迁移到 `MySQL`
- 社区内容、审核、审计、配置迁移到 `MongoDB`
- `BMFS` 接管状态流转
- 管理端和客户端切到新契约
- 删除旧 `postgres/review_status/legacy service` 代码

## 工作流

### 工作流 A：存储职责重划

- 核心身份、角色、组织、绑定关系使用 `MySQL`
- 社区内容、门户内容、通知内容、审核单、审计与配置使用 `MongoDB`
- 会话、验证码、热点缓存继续使用 `Redis`
- 同步明确 `MySQL/GORM` 查询规范与 `MongoDB` 查询 builder 规范

退出条件：

- 每个模块都有明确主事实源
- 不再出现“同一类数据既想当事务核心又想当灵活文档”的模糊边界

### 工作流 B：社区模型重建

- 重建统一 `community_content` 聚合
- 将跑腿、二手、资料、失物招领、拼车、组局收敛为“公共字段 + 类型载荷”
- 独立设计 `moderation_cases`、`state_transition_logs`、`feed_projections`
- 不做历史数据迁移，不保留旧文档结构兼容层
- 新模型落地后直接删除旧社区模型代码，不维护新旧并存

退出条件：

- 社区模型不再只是旧表结构平移
- 新内容类型接入不需要再复制一整套状态字段和列表逻辑

### 工作流 C：BMFS 接入

- 为六类社区内容定义单一 `status` 生命周期
- 所有 `can_xxx` 从 `BMFS` 派生
- 审核阶段并入主状态机，不额外抽 `review_status`
- 客户端不得直接提交目标状态，只能提交动作命令

退出条件：

- `service` 不再散落大量状态判断
- 多端直接消费统一状态语义

### 工作流 D：读模型与审计迁移

- 审核列表、首页流、审计日志、门户聚合改为面向 `MongoDB` 读模型
- 关键操作统一写入审计与状态迁移日志

退出条件：

- 社区、审核、审计、配置可以按聚合读取，不需要大量跨表拼装

### 工作流 E：契约与客户端迁移

- 先改 `OpenAPI`
- 再生成 `JS/Dart SDK`
- 最后切客户端消费；不保留旧 DTO 兼容层
- 客户端切换完成后，删除旧 handler/service/repo 中的历史分支逻辑
- 切换前完成 SQL/NoSQL 注入、越权和限流回归

退出条件：

- 客户端不再耦合旧 DTO
- 客户端只消费新状态语义与权限结果

## 风险

- 双存储会增加运维复杂度
- `BMFS` 若无现成成熟 Go 适配层，初期需要额外封装
- 社区模型重建会影响审核、首页流、通知、配置多个模块
- 若不严格执行“先契约后实现”，多端会再次漂移
- 单一 `status` 设计如果枚举边界不清，会把内容类型差异重新塞回代码分支

## 风险应对

- 先文档冻结，再做代码迁移
- 不保留旧模型兼容层，直接围绕新聚合和新契约实现
- 关键状态切换先补自动化测试，再切生产实现
- 使用执行计划明确分阶段退出条件，避免“大爆炸切换”
- 删除旧逻辑前先完成新链路最小可运行闭环，避免删完后无可用路径
