# 0002 M1/M2 联调与收口计划

## 目标

在 `0001` 已完成基础骨架的前提下，把当前仓库从“后端主通路已具备、前端部分真实联调、部分页面仍为演示态”推进到“V1 主线可联调、可验收、可回归”。

## 背景

- 后端 `iam`、`campus_life`、`portal`、`notification`、`analytics` 已具备基础能力，但前端消费并未全部收口。
- 管理后台已完成工程和部分真实接口接入，仍存在若干本地样例数据页面。
- 微信小程序核心校园生活链路已跑通，但门户、通知、消息中心、教务接口对齐仍未完成。
- 经当前轮次确认，`Flutter App` 外壳与 `academic` 真实校园连接器演进统一放到 `P2`，不阻塞当前 V1 主线收口。

## 优先级

### P0：直接影响 V1 联调与验收

- 完成 `packages/contracts` 到客户端消费层的最小闭环，明确 OpenAPI 变更与 SDK 生成流程。
- 收口 `apps/admin-web` 中仍为本地演示态的页面，统一切到真实后端接口。
- 收口 `apps/miniapp-wechat` 的门户、通知、消息中心与旧教务接口路径对齐。
- 修正文档与计划中的完成度表述，确保文档反映当前事实而不是目标状态。

### P1：保证交付稳定性与回归效率

- 建立最小 CI，覆盖后端测试、后台类型检查/构建、小程序关键校验。
- 补齐 lint / format / test 的统一入口，减少跨端回归成本。
- 继续完善审计、指标、追踪与持久化模式下的联调一致性。

### P2：延后但明确保留的工作

- 初始化 `apps/mobile-flutter` 外壳，并补齐 `feature-first + MVVM + repository` 基线。
- 将 `academic` 从 mock 联调能力演进到真实连接器、同步任务与本地快照。
- 评估聊天等非 V1 主线页面的去留，避免继续消耗主线收口资源。

## 工作流

当前进展（截至 `2026-05-12`）：

- 已收口 `portal` 基础域的管理端真实能力，补齐轮播 CRUD 与公告更新/删除接口，并同步更新 `OpenAPI` 契约。
- 已将 `admin-web` 的轮播管理、公告管理、审核历史切到真实接口；文章页改为预留入口，不再继续请求不存在的后端能力。
- 已修复 `iam/repo` 的既有失败用例，`services/api-server` 全量 `go test ./...` 已恢复通过。
- 已将 `admin-web` 的校园生活管理页切到真实审核分页数据，并补齐列表详情与上下线/重新发布操作，不再依赖本地样例数组。

### 工作流 A：契约与 SDK 收口

- 盘点当前小程序与后台手写 API 封装的契约漂移点。
- 明确 `JS SDK` 与 `Dart SDK` 的生成目录、命令入口、版本更新方式。
- 至少为后续前端改造建立“改契约 -> 生成 SDK -> 消费”的固定流程。

### 工作流 B：管理员后台真实联调

- 将轮播管理切换为真实列表、创建、编辑、删除接口。
- 将审核历史切换为真实审核结果查询，而不是前端本地数组。
- 将仍使用本地样例数据的校园生活管理页逐步切到后端数据源。
- 校验权限码、路由守卫、按钮可见性是否与服务端能力一致。

当前状态：

- 轮播管理、公告管理、待审核、审核历史、校园生活管理主线页面均已接入真实接口。
- 登录、路由守卫与侧栏展示已切到后端返回的真实权限码。
- 剩余问题集中在小程序主线尚未收口，以及部分 IAM 页面本身仍是演示数据。

### 工作流 C：微信小程序主线收口

- 接入门户首页、公告、通知与未读数。
- 将消息中心从占位页升级为真实列表页，明确是否继续保留聊天入口。
- 对齐教务读取接口，清理旧路径和页面中的演示数据依赖。
- 继续校验“未绑定教务不可见联系方式”的端到端语义。

### 工作流 D：质量与运维基线

- 建立仓库级 CI 入口和最小通过标准。
- 为后端与前端分别补齐可执行的本地校验命令。
- 梳理 `memory / postgres / 本地样例数据` 的适用边界，减少联调偏差。

### 工作流 E：P2 延后事项

- `Flutter App` 外壳初始化与主线能力接入。
- `academic` 真实校园系统接入、同步与缓存策略。

## 验收标准

### P0 验收

- 管理后台不再依赖本地数组维护轮播、审核历史和主线校园生活列表。
- 小程序可以真实读取门户、通知、消息中心数据，不再仅展示“开发中”占位。
- 小程序教务相关接口路径与后端当前契约一致，不再继续扩散旧接口语义。
- 当前计划文档能明确区分“已联调完成”和“仍待收口”的范围。

### P1 验收

- 至少存在一套可重复执行的 CI 校验入口。
- 开发者能在仓库内快速完成后端、后台、小程序的最小自检。

### P2 验收

- `Flutter App` 拥有最小可运行工程和共享业务语义接入基线。
- `academic` 不再只依赖 mock provider 承担真实校园系统接入预期。

## 风险

- 前端页面现状与文档状态不一致，容易误判剩余工作量。
- 小程序历史接口路径与当前后端契约存在漂移，修正时可能影响已有页面。
- 管理后台多个页面若同时从演示态切换到真实接口，容易暴露权限和空态处理缺口。
- CI 接入过晚会放大跨端改动的回归成本。

## 风险应对

- 所有状态修正先落文档，再安排跨端收口。
- 先优先接主线读取与管理功能，再处理非主线页面细节。
- 每次完成一类页面收口后立即补自检命令与计划状态。
- `Flutter` 与真实教务接入保持明确记录，但不插队阻塞当前 `P0 / P1`。

## 退出条件

- `P0` 项全部完成，仓库进入“主线真实联调完成”状态。
- `P1` 至少完成最小 CI 与统一校验入口。
- `P2` 明确保留在计划中，不因当前收口而丢失上下文。
