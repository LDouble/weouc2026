# V1 后端业务逻辑补齐 PRD

## 背景

`services/api-server` 已完成基础工程、`IAM` 持久化，以及小程序主链路的第一批校园生活接口，但距离 V1 产品范围仍有明显缺口：

- `campus_life` 虽已切到可配置的 `memory / postgres` 双后端，但 `meetup/carpool` 与统一审核闭环仍未落地
- `meetup` 组局能力尚未落地
- `portal`、`notification`、`file_center`、`analytics` 仍是预留模块
- 管理端所需的审核、运营、统计与权限闭环尚未形成

用户提出“继续完成开发所有的业务逻辑”，因此需要先把剩余范围收敛为分阶段可执行计划，再按阶段逐轮提交。

## 目标

在不破坏现有模块化单体边界的前提下，补齐 V1 后端剩余核心业务逻辑，使后端具备以下能力：

- 校园生活五类信息具备持久化、发布、浏览、详情、基础状态流转、后台审核闭环
- 门户内容、公告、轮播具备管理与投放能力
- 站内消息/通知具备最小发送与查询能力
- 文件上传元数据与附件引用具备统一入口
- 管理员具备基础审核、权限校验、数据看板和审计能力

## 非目标

- 不接入真实微信开放平台和真实教务系统
- 不实现复杂工作流引擎或即时聊天
- 不在本任务内推进完整教务能力，如课表、成绩、考试
- 不在单轮提交内一次性完成全部模块；必须按阶段独立提交

## 设计约束

- 业务规则只在后端 `service` 层落地
- 数据读写必须经 `repo` 层，外部系统必须经 `providers`
- 所有新接口先更新 `packages/contracts/openapi/api-server.yaml`
- 管理端权限裁决必须在服务端完成，不能依赖前端隐藏按钮
- 联系方式、审核状态、可见性等跨端一致语义必须继续由后端统一裁剪

## 当前现状判断

### 已完成

- `system`：健康检查、就绪检查、会话画像
- `iam`：微信登录、资料绑定、`PostgreSQL + Redis` 持久化
- `campus_life`：二手、跑腿、资料、失物招领的基础读写接口、关键交互，以及 `memory / postgres` 双后端
- `file_center`：腾讯 COS 临时上传凭证、下载预签名与文件路径引用

### 主要缺口

- `campus_life`：
  - 缺少 `meetup` 组局能力
  - 缺少统一审核状态与后台管理入口
- `portal`：未落地
- `notification`：未落地
- `analytics`：未落地
- 管理端运营/审核/统计所需接口：未落地

## 分阶段范围

### 阶段 1：校园生活持久化与组局补齐

目标：把现有校园生活主链路从“种子演示数据”推进到“可持续运行的持久化业务数据”。

范围：

- 把 `campus_life` 从内存仓储迁移到 `PostgreSQL`
- 为二手、跑腿、资料、失物招领建立表结构与迁移
- 补齐 `meetup` 组局模块
- 为校园生活五类内容统一补充：
  - 发布状态
  - 审核状态
  - 发布人/接单人/参与人
  - 创建/更新时间
- 保持现有小程序接口兼容，必要时增补管理端接口

退出条件：

- 服务重启后校园生活运行期数据不丢失
- 五类内容都至少具备“发布、列表、详情、基础状态流转”

### 阶段 2：门户内容与文件中心

目标：补齐首页运营所需的数据源和统一附件语义。

范围：

- 落地 `portal`
  - 公告
  - 轮播
  - 资讯/文章
- 落地 `file_center`
  - 文件元数据
  - 附件引用关系
  - 上传票据接口占位或 mock 直传语义
- 打通 `portal` 与 `campus_life` 的封面/附件引用

退出条件：

- 门户内容可管理、可投放、可查询
- 附件与文件引用不再散落在业务表中

### 阶段 3：通知、审核、统计与管理端能力

目标：形成 V1 最小管理闭环。

范围：

- 落地 `notification`
  - 站内消息
  - 发送记录
  - 用户消息列表/已读状态
- 落地 `analytics`
  - 审计日志
  - 基础运营统计
  - 管理端看板聚合接口
- 管理端审核能力
  - 内容审核列表
  - 审核通过/驳回
  - 基础用户治理
- 扩展 `IAM` 角色权限模型到管理员语义

退出条件：

- 管理员可查看待审内容并完成审核
- 用户可查询自己的站内消息
- 管理端可拉取最小统计数据

## 建议实施顺序

推荐先做 **阶段 1**，原因：

- 当前最大风险是 `campus_life` 仍为内存仓储，任何继续叠加业务都会建立在易失状态之上
- `meetup` 属于产品 V1 明确范围，且和现有校园生活模型最接近，复用性最高
- 完成阶段 1 后，后续 `portal`、`notification`、管理端审核都能建立在稳定内容主数据之上

## 验收标准

- `go test ./...` 持续通过
- 新增模块具备契约、实现、仓储迁移和中文文档
- 关键主链路具备自动化测试
- 每完成一个阶段，必须形成独立 commit，并同步更新 `docs/PLANS.md` 与相关 README

## 风险

- “完成所有业务逻辑” 若不切阶段，会导致单轮改动过大、回归面失控
- 在 `campus_life` 持久化前继续叠加后台能力，会放大后续迁移成本
- 文件中心与通知中心若先行落地，容易因为内容主数据不稳定而返工
