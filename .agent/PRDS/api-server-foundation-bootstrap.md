# 后端服务器基础工程初始化 PRD

## 背景

当前仓库已经完成总体架构、产品范围和执行计划文档，但 `services/api-server` 仍处于空骨架状态。多端联调、契约沉淀和后续业务域实现都依赖一个可运行、可测试、可持续扩展的后端基础工程。

## 目标

在不提前承诺具体业务功能闭环的前提下，完成后端第一阶段底座建设，使 `services/api-server` 具备以下能力：

- 以 `Go + Gin` 启动可运行 HTTP 服务
- 建立符合 `types -> config -> repo -> service -> runtime -> transport` 的目录骨架
- 提供统一配置加载、结构化日志、错误码与错误响应
- 提供基础鉴权中间件和权限上下文装配能力
- 提供健康检查、就绪检查、版本信息等系统级接口
- 提供最小可维护的 OpenAPI 基线文档
- 提供最基础的单元测试与运行说明

## 非目标

- 不在本轮实现跑腿、组局、二手、资料、失物招领等具体业务域闭环
- 不在本轮打通 PostgreSQL、Redis、对象存储等真实依赖
- 不在本轮接入真实微信登录、教务绑定、短信、对象存储 provider
- 不在本轮生成前端 SDK

## 设计约束

- 必须遵守模块化单体分层方向，不允许跨层反向依赖
- 对外接口先定义契约，再实现路由
- 权限裁决入口必须预留在后端，不把业务权限逻辑下放给客户端
- 外部系统能力只允许通过 `providers` 层暴露
- 代码、文档、计划必须同步更新

## 范围

### 1. 工程骨架

- 初始化 `go.mod`
- 建立 `cmd/`、`internal/modules/`、`internal/platform/`、`internal/providers/`、`api/openapi/`、`migrations/` 等目录
- 建立至少一个示例模块骨架，验证分层规范可落地

### 2. 平台基础能力

- 环境变量配置加载
- 结构化日志封装
- 统一错误码、错误响应、请求链路标识
- Gin 路由装配和全局中间件

### 3. 鉴权与权限上下文

- 基于请求头注入用户身份上下文
- 支持角色、权限码、教务绑定状态等基础字段
- 提供需要登录、需要权限的中间件能力

### 4. 系统接口

- `GET /healthz`
- `GET /readyz`
- `GET /api/v1/system/profile`

### 5. 契约与测试

- 为系统接口补齐 OpenAPI 文档
- 为核心基础能力补齐单元测试

## 验收标准

- 能通过命令启动本地 HTTP 服务
- 系统接口可正常返回统一结构
- 未登录、缺少权限、业务错误都能返回统一错误格式
- 目录结构能清晰体现分层和模块边界
- OpenAPI 文档与实现一致
- 至少存在一组自动化测试覆盖基础能力

## 风险

- 若过早写入业务细节，后续契约和模块边界容易返工
- 若基础错误码和权限上下文设计过弱，后续多端联调成本会明显上升
- 若不先建立最小契约文档，客户端接口很快会漂移

## 交付物

- `services/api-server` 可运行基础工程
- 配套的运行说明与模块说明
- 对应计划与文档更新
