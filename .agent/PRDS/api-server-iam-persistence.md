# IAM 持久化落地 PRD

## 背景

`api-server` 当前已经具备微信登录、资料查询、教务绑定等 IAM 主链路接口，但底层仍使用内存仓储。这样虽然便于联调，却会导致服务重启后 token、验证码、绑定信息全部丢失，也让上一轮接入的 `PostgreSQL`、`Redis` Docker 中间件尚未真正进入业务链路。

## 目标

在不改变现有接口语义的前提下，把 IAM 模块的关键状态改为持久化存储：

- `PostgreSQL`：用户基础信息、角色权限、教务绑定资料
- `Redis`：登录会话 token、教务验证码
- 服务启动时执行最小迁移，避免本地 Docker 环境需要手工建表

## 非目标

- 不在本轮把 `campus_life` 业务数据迁移到 `PostgreSQL`
- 不接入真实微信、真实教务系统
- 不引入完整第三方迁移框架或生产级分布式锁
- 不调整当前 OpenAPI 接口语义

## 设计约束

- 业务规则仍只保留在 `service` 层，仓储只负责存取
- `handler` / `auth middleware` 不能直接访问数据库或 Redis
- 配置必须显式决定 IAM 使用内存还是持久化后端，避免因为依赖启用状态产生隐式行为
- 若配置为持久化后端但中间件或迁移失败，服务应启动失败而不是悄悄降级

## 本轮范围

### 1. 配置与装配

- 新增 IAM 存储后端配置
- 新增自动迁移开关
- 在 `app` 装配层根据配置选择 `memory` 或 `postgres_redis`

### 2. 数据持久化

- 用户表：`openid`、昵称、头像、角色、权限、教务绑定资料、时间戳
- Redis 会话：token 到 session 的映射，按过期时间设置 TTL
- Redis 验证码：按学号存储验证码票据，按过期时间设置 TTL

### 3. 迁移执行

- 维护 `schema_migrations`
- 服务启动时按顺序执行未应用的 SQL 迁移
- 先提供 IAM 最小表结构

### 4. 测试与文档

- 补充配置校验、迁移与仓储基础测试
- 更新 `README`、计划和运维说明

## 验收标准

- 配置为 `postgres_redis` 时，微信登录后重启服务，用户绑定资料不会因为进程重启丢失
- token 和验证码由 `Redis` 承载，并遵循 TTL 过期
- 新库环境可通过服务启动自动完成 IAM 所需建表
- `go test ./...` 通过

## 风险

- 若仓储接口仍沿用“无错误返回”，真实存储故障会被吞掉，必须收敛为显式错误
- 自动迁移逻辑若处理不当，后续扩展多模块迁移时容易失控，因此本轮应保持实现简单且可审计
- `campus_life` 仍是内存仓储，登录与资料状态持久化后，业务发布数据仍然会在重启后丢失，这一点需要在文档中明确
