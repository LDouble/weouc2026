# 小程序 API 对接与联调 PRD

## 背景

`apps/miniapp-wechat` 当前已经完成小程序壳层重构，并接入了登录、我的页、首页动态、二手、跑腿、资料、失物招领等第一批接口封装与服务层映射。但从现状看，前后端仍未完成真正的“业务闭环联调”：

- 小程序当前主要完成了列表页和部分绑定资料流
- 详情页、发布页、上传链路、错误态和重登恢复仍需要与后端能力逐一对齐
- `carpool/meetup`、消息页、资料上传、统一发布等页面仍存在 mock 或占位实现
- 后端后续还会继续补齐 `campus_life` 持久化与组局等能力，小程序需要按阶段跟进契约变化

因此需要单独为“小程序 API 对接与联调”建立任务与 PRD，而不是混在后端总任务中。

## 目标

在保持小程序现有分层约束的前提下，完成小程序与 `api-server` 的 API 对接和联调，使以下链路可真实跑通：

- 微信登录与会话恢复
- 教务绑定与解绑
- 首页动态流
- 二手、跑腿、资料、失物招领的列表、详情、发布
- 小程序对 `401/404/业务错误/网络错误` 的统一处理
- 页面跳转、发布成功回流、详情可见性与联系方式裁剪的端到端验证

## 非目标

- 不在本任务内实现真实微信开放平台登录
- 不在本任务内实现完整聊天系统
- 不在后端契约尚未稳定前强行接入未落地模块
- 不把页面业务规则下沉到页面层，仍必须经 `services/` 和服务端裁决

## 设计约束

- `api/` 只描述接口，不承载页面语义
- `services/` 负责 DTO 到页面模型映射
- 页面只消费服务层结果，不自行推断权限或联系方式可见性
- 所有联调以 `packages/contracts/openapi/api-server.yaml` 和后端实际返回为准
- 后端接口调整后，小程序必须同步更新 `API.md` 和相关服务层

## 当前现状

### 已接通

- `wx.login -> /auth/wechat/login`
- `GET/POST/PUT /student`
- `POST /edu/send-captcha`
- `GET /feed/list`
- `GET /market/list`
- `GET /errand/list`
- `GET /resource/list`
- `GET /lostFound/list`
- 部分详情页和发布页封装已存在

### 主要缺口

- 详情页联调不完整，部分页面仍未真实消费后端详情结构
- 发布页依赖上传接口，但 `upload` 仍是占位/mock 语义
- `carpool/meetup` 页面仍是本地数据或占位
- 消息页当前没有真实后端通知接口支撑
- 错误态、空态、会话恢复和重试策略需要按真实后端结果收敛

## 分阶段范围

### 阶段 A：当前已落地后端能力的全面对接

范围：

- 校验并修正 `api/request.js` 对标准响应包裹结构的适配
- 登录、会话恢复、我的页、教务绑定页与后端现状对齐
- 二手、跑腿、资料、失物招领列表与详情页真实联调
- 发布页与当前后端发布接口对齐
- 小程序 `API.md` 与实际后端契约同步

退出条件：

- 当前已存在的后端接口都能被小程序真实消费
- 登录、绑定、列表、详情、发布链路无本地假数据依赖

### 阶段 B：跟进后端新增业务能力

范围：

- 跟进 `campus_life` 持久化后的字段调整
- 接入 `meetup/carpool` 真实接口
- 接入门户内容、消息中心等后续后端能力
- 收敛统一发布页与不同业务域发布入口的参数差异

退出条件：

- 小程序对后端新增业务域具备真实接口消费能力

### 阶段 C：联调验证与回归

范围：

- 真机/开发者工具环境联调
- 登录失效、网络异常、未绑定教务、联系方式不可见等场景回归
- 发布成功后的回流、详情页回跳、我的发布/我的接单页回归

退出条件：

- 关键用户路径可以连续跑通
- 错误提示和降级路径明确

## 建议实施顺序

推荐先做 **阶段 A**，原因：

- 当前小程序已经具备一批接口封装，最容易先清理掉假数据和结构不一致问题
- 后端后续即使继续扩展，阶段 A 产物也不会浪费
- 只有先把现有链路跑通，后续才有条件继续做 `meetup`、消息和管理回流

## 验收标准

- 小程序核心路径可通过真实 `api-server` 跑通：登录、教务绑定、首页动态、二手/跑腿/资料/失物招领的列表、详情、发布
- 页面不再依赖本地静态 `data.js` 作为真实业务源
- `401` 自动恢复、失败跳转登录、业务错误提示行为稳定
- 小程序接口说明文档与后端契约同步
- 每一轮对接改动都形成独立 commit

## 风险

- 后端总任务尚未全部完成，如果一次性追所有页面，联调目标会不断漂移
- `upload`、`notification`、`meetup` 仍依赖后端后续实现，不适合和当前已落地链路混做
- 小程序存在部分历史页面和本地 `data.js`，如果不先清理数据来源，联调结果会失真
